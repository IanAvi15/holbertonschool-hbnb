# HBnB - Part 2: Implementation of Business Logic and API Endpoints

## Description

This is Part 2 of the HBnB Project. It focuses on setting up the initial modular project structure for the application, following a layered architecture composed of the Presentation, Business Logic, and Persistence layers. The project uses Flask and Flask-RESTx to expose a RESTful API, and adopts the Facade design pattern to manage communication between layers cleanly.

At this stage, the goal is to establish a solid, scalable foundation: the folder structure, the Flask application factory, an in-memory repository for object storage, and the Facade class that will mediate access to that storage. Business logic, full CRUD endpoints, and database persistence will be built out in later tasks.

### Directory Overview

- **app/** — the core application package.
- **app/api/v1/** — API endpoints, versioned under `v1`. Each resource (users, places, reviews, amenities) has its own module; routes will be implemented in upcoming tasks.
- **app/models/** — business logic classes representing the core entities (`User`, `Place`, `Review`, `Amenity`).
- **app/services/** — houses the Facade pattern implementation (`HBnBFacade`), which coordinates communication between the API layer and the persistence layer. A singleton `facade` instance is created in `services/__init__.py`.
- **app/persistence/** — the storage layer. Currently implemented as an in-memory repository (`InMemoryRepository`) conforming to an abstract `Repository` interface; this will be replaced with a SQLAlchemy-backed implementation in Part 3.
- **run.py** — entry point for running the Flask development server.
- **config.py** — environment-specific configuration settings (e.g., debug mode, secret key).
- **requirements.txt** — Python package dependencies.

## Business Logic Layer

The core entities live in `app/models/` and all inherit from `BaseModel` (`app/models/base_model.py`), which provides a UUID `id`, `created_at`/`updated_at` timestamps, a `save()` method to refresh `updated_at`, and an `update(data)` method to bulk-update attributes from a dictionary.

### Entities

- **User** (`user.py`) — `first_name`, `last_name`, `email`, `is_admin`. Validates name lengths (max 50 chars) and basic email format. Tracks the places it owns and the reviews it has written.
- **Place** (`place.py`) — `title`, `description`, `price`, `latitude`, `longitude`, `owner`. Validates title length, positive price, and latitude/longitude ranges. Tracks its own list of `reviews` and `amenities`.
- **Review** (`review.py`) — `text`, `rating` (1–5), `place`, `user`. Validates rating range and that `place`/`user` are real instances of those classes.
- **Amenity** (`amenity.py`) — `name` (max 50 chars).

### Relationships

- A `User` can own multiple `Place` instances (one-to-many). Creating a `Place` automatically registers it on the owner's `places` list.
- A `Place` can have multiple `Review` instances (one-to-many). Creating a `Review` automatically registers it on both the place's `reviews` list and the reviewing user's `reviews` list.
- A `Place` can have multiple `Amenity` instances (many-to-many, simplified as a list). Use `place.add_amenity(amenity)` to associate one.

### Usage Example

```python
from app.models.user import User
from app.models.place import Place
from app.models.review import Review
from app.models.amenity import Amenity

owner = User(first_name="Alice", last_name="Smith", email="alice@example.com")
place = Place(title="Cozy Apartment", description="A nice place to stay",
              price=100, latitude=37.7749, longitude=-122.4194, owner=owner)

review = Review(text="Great stay!", rating=5, place=place, user=owner)
place.add_review(review)

wifi = Amenity(name="Wi-Fi")
place.add_amenity(wifi)

place.update({"price": 120})  # updates price and refreshes updated_at
```

### Note on Email Uniqueness

The `User` class validates email *format* on its own, but it has no way to check *uniqueness* in isolation — that requires knowledge of all other users. Uniqueness should be enforced at the repository/facade level (e.g., calling `get_by_attribute('email', email)` before creating a new user) rather than inside the `User` class itself.

## User Endpoints

Implemented in `app/api/v1/users.py`, registered under `/api/v1/users`.

| Method | Endpoint | Description | Success | Errors |
|---|---|---|---|---|
| POST | `/api/v1/users/` | Register a new user | 201 | 400 (duplicate email or invalid input) |
| GET | `/api/v1/users/` | List all users | 200 | — |
| GET | `/api/v1/users/<user_id>` | Get a user by ID | 200 | 404 (not found) |
| PUT | `/api/v1/users/<user_id>` | Update a user by ID | 200 | 404 (not found), 400 (duplicate email or invalid input) |

Note: DELETE is intentionally not implemented for users in this part of the project.

### Example

```
curl -X POST http://localhost:5000/api/v1/users/ \
  -H "Content-Type: application/json" \
  -d '{"first_name": "John", "last_name": "Doe", "email": "john.doe@example.com"}'
```

```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "first_name": "John",
  "last_name": "Doe",
  "email": "john.doe@example.com"
}
```

All endpoints route through `HBnBFacade` (`app/services/facade.py`), which now implements `create_user`, `get_user`, `get_user_by_email`, `get_all_users`, and `update_user` against the in-memory `user_repo`.

On `PUT`, if the request changes the email to one already used by a *different* user, the API returns `400 {"error": "Email already registered"}` — this mirrors the uniqueness check already enforced on `POST` so the same email can't be assigned to two users via an update.

## Place Endpoints

Implemented in `app/api/v1/places.py`, registered under `/api/v1/places`.

| Method | Endpoint | Description | Success | Errors |
|---|---|---|---|---|
| POST | `/api/v1/places/` | Register a new place | 201 | 400 (invalid input, unknown `owner_id`, or unknown amenity id) |
| GET | `/api/v1/places/` | List all places (summary view) | 200 | — |
| GET | `/api/v1/places/<place_id>` | Get full place details, including owner and amenities | 200 | 404 (not found) |
| PUT | `/api/v1/places/<place_id>` | Update a place by ID (partial updates allowed) | 200 | 404 (not found), 400 (invalid input) |

Note: DELETE is intentionally not implemented for places in this part of the project, and reviews are handled in a separate task/endpoint.

### Creating a Place

`POST` takes `owner_id` (a `User` id) and a list of `amenities` (a list of `Amenity` ids) — not full nested objects. The facade resolves these against `user_repo` and `amenity_repo`:

```
curl -X POST http://localhost:5000/api/v1/places/ \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Cozy Apartment",
    "description": "A nice place to stay",
    "price": 100.0,
    "latitude": 37.7749,
    "longitude": -122.4194,
    "owner_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "amenities": ["4fa85f64-5717-4562-b3fc-2c963f66afa6"]
  }'
```

If `owner_id` doesn't match an existing user, or any id in `amenities` doesn't match an existing amenity, the request fails with `400` and a descriptive error message rather than creating a partial/invalid place.

`description` is optional (defaults to an empty string) — only `title`, `price`, `latitude`, `longitude`, and `owner_id` are required.

### Retrieving Places

- `GET /api/v1/places/` returns a **summary** for each place: `id`, `title`, `latitude`, `longitude` (matches the list example in the spec).
- `GET /api/v1/places/<place_id>` returns the **full detail** view, with the owner and amenities fully expanded into nested objects (not just ids):

```json
{
  "id": "1fa85f64-5717-4562-b3fc-2c963f66afa6",
  "title": "Cozy Apartment",
  "description": "A nice place to stay",
  "price": 100.0,
  "latitude": 37.7749,
  "longitude": -122.4194,
  "owner": { "id": "...", "first_name": "John", "last_name": "Doe", "email": "..." },
  "amenities": [{ "id": "...", "name": "Wi-Fi" }]
}
```

### Updating a Place

`PUT` supports **partial updates** — you only need to send the fields you want to change (matching the spec's example, which only sends `title`, `description`, and `price`). `owner_id` and `amenities` may also be sent to reassign the owner or replace the full amenities list; both are still validated against the repositories. On success, the response is a simple confirmation message, not the updated object:

```json
{
  "message": "Place updated successfully"
}
```

### Note on Price Validation

The `Place` model enforces a strictly positive price (`price > 0`), carried over from when it was first implemented. This task's wording calls for a "non-negative" price (which would also allow `0`) — if your checker expects `price = 0` to be accepted, the bound in `app/models/place.py`'s `_validate_price` would need to change from `value <= 0` to `value < 0`.

The `HBnBFacade` (`app/services/facade.py`) now implements `create_place`, `get_place`, `get_all_places`, and `update_place`, handling the owner/amenity id-to-object resolution described above.

## Review Endpoints

Implemented in `app/api/v1/reviews.py`, registered under `/api/v1/reviews`. A related endpoint also lives under the places namespace, per the spec's instruction to keep it there.

| Method | Endpoint | Description | Success | Errors |
|---|---|---|---|---|
| POST | `/api/v1/reviews/` | Register a new review | 201 | 400 (invalid input, unknown `user_id`, or unknown `place_id`) |
| GET | `/api/v1/reviews/` | List all reviews (summary view) | 200 | — |
| GET | `/api/v1/reviews/<review_id>` | Get a review by ID (full view) | 200 | 404 (not found) |
| PUT | `/api/v1/reviews/<review_id>` | Update a review by ID (partial updates allowed) | 200 | 404 (not found), 400 (invalid input) |
| DELETE | `/api/v1/reviews/<review_id>` | Delete a review by ID | 200 | 404 (not found) |
| GET | `/api/v1/places/<place_id>/reviews` | List all reviews for a specific place (summary view) | 200 | 404 (place not found) |

Reviews are the only entity in this part of the project for which DELETE is implemented.

### Creating a Review

`POST` takes `user_id` and `place_id` — not full nested objects. The facade resolves both against `user_repo` and `place_repo` before constructing the `Review`:

```
curl -X POST http://localhost:5000/api/v1/reviews/ \
  -H "Content-Type: application/json" \
  -d '{
    "text": "Great place to stay!",
    "rating": 5,
    "user_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "place_id": "1fa85f64-5717-4562-b3fc-2c963f66afa6"
  }'
```

If either id doesn't correspond to an existing user/place, the request fails with `400` rather than creating an orphaned review.

### Two Response Shapes for Reviews

- `GET /api/v1/reviews/` and `GET /api/v1/places/<place_id>/reviews` return a **summary**: `id`, `text`, `rating` only (matching the spec's list examples).
- `GET /api/v1/reviews/<review_id>` and the `POST` response return the **full** shape, including `user_id` and `place_id`.

### Updating and Deleting

`PUT` supports partial updates (the spec's example only sends `text` and `rating`), and returns a confirmation message rather than the updated object:

```json
{
  "message": "Review updated successfully"
}
```

`DELETE` returns a similar confirmation:

```json
{
  "message": "Review deleted successfully"
}
```

Deleting a review also removes it from the associated place's and user's in-memory `reviews` lists, so a deleted review won't linger in `GET /api/v1/places/<place_id>` or in the place's nested `reviews` field.

### Place Detail Now Includes Reviews

Per this task, `GET /api/v1/places/<place_id>` now also returns a `reviews` list (each entry: `id`, `text`, `rating`, `user_id`), alongside the existing `owner` and `amenities` fields.

### A Note on the Place Model in `places.py`

The spec's review-endpoint instructions update `place_model` in `places.py` to add `owner` (nested), `amenities` (as nested objects rather than ids), and `reviews` (nested) fields — useful for *documenting the output shape*, but incompatible with using that same model to *validate POST/PUT input*, where `amenities` must be a flat list of ids and `owner`/`reviews` shouldn't be expected from the client at all. To resolve this, `place_model` stays as the spec describes (used for Swagger documentation), and a separate `place_input_model` — kept as a flat list of amenity ids — is used in the `@api.expect(...)` decorators for `POST`/`PUT` on places. This avoids rejecting the spec's own example payloads while still documenting the richer output shape.

### Note on Validated Updates

The base `update()` method (inherited from `BaseModel`) does a raw `setattr` for any key passed in, with no validation. For `Review`, this has been overridden so that `update()` re-validates `text`, `rating`, `place`, and `user` before applying changes — so a `PUT` with an out-of-range rating (e.g. `99`) correctly returns `400` instead of silently corrupting the stored review.

The `HBnBFacade` (`app/services/facade.py`) now implements `create_review`, `get_review`, `get_all_reviews`, `get_reviews_by_place`, `update_review`, and `delete_review`.

### A note on test isolation

`HBnBFacade` is a module-level singleton, so its in-memory repositories persist across separate `create_app()` calls within the same Python process. `test_api.py` explicitly resets the facade's repositories in `setUp()` to keep tests isolated from one another — see `TESTING_REPORT.md` for the full story on why this was necessary.
