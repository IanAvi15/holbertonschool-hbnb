## Overview

HBnB Evolution is a simplified AirBnB-like platform supporting user management, property listing, reviews, and amenity tracking. The application follows a **three-layer architecture** separated by concern:

| Layer | Responsibility |
|---|---|
| Presentation | Expose REST API endpoints; validate HTTP requests |
| Business Logic | Enforce business rules; model domain entities |
| Persistence | Store and retrieve data from the database |

Layers communicate exclusively through a **Facade** — a single interface that decouples callers from implementation details of the layer below.

### Layer descriptions

**Presentation layer** handles all user-facing interactions. It parses incoming HTTP requests, validates input format (not business rules), authenticates tokens, and serializes responses. It never touches the database directly.

**Business logic layer** contains the four core domain models and all application rules — ownership checks, duplicate-email prevention, rating constraints, and so on. This is the only layer that "knows" what HBnB is for. It exposes its capabilities through a Facade object, keeping the presentation layer free of domain knowledge.

**Persistence layer** owns all database interactions. Repositories provide `find`, `save`, `update`, and `delete` operations for each entity. The ORM translates between Python objects and SQL rows. In Part 3, this layer will be implemented with SQLAlchemy backed by SQLite (development) or PostgreSQL (production).

### Facade pattern

The Facade is a single service class (or module) that acts as the entry point into the business logic. The presentation layer calls only Facade methods — it never instantiates domain models or queries the database directly. This keeps the API thin and makes the business logic independently testable.

### Entity summaries

**BaseModel** — abstract parent. Generates a UUID on instantiation, timestamps `created_at` and `updated_at` automatically, and provides `save()` (persist/flush) and `to_dict()` (serialization for API responses).

**User** — represents both regular visitors and administrators (`is_admin: bool`). Passwords must be stored hashed, never in plain text. An email must be unique across all users.

**Place** — a property listing owned by one `User`. A place can be associated with many amenities (many-to-many). Latitude/longitude must be validated to realistic ranges.

**Review** — ties a `User` to a `Place` with a 1–5 rating and free-text comment. One user may leave at most one review per place.

**Amenity** — a simple named feature (e.g. "Wi-Fi", "Pool"). Managed centrally and associated to places through a join table.

### 3.1 User Registration

**Endpoint:** `POST /api/users`

```
Client          API              Facade           User model        DB
  │               │                │                  │              │
  │─POST /api/users──────────────►│                  │              │
  │               │─validate_and_register(data)──►  │              │
  │               │                │─find_by_email(email)─────────►│
  │               │                │◄─────────────── None ─────────│
  │               │                │─User(data)──────────────────► │
  │               │                │                  │─INSERT──── ►│
  │               │                │                  │◄──OK────── │
  │               │                │◄─────user.to_dict()───────────│
  │               │◄──{id, email}──│                  │              │
  │◄─201 Created──│               │                  │              │
  │               │                │                  │              │
  │[alt] email already registered  │                  │              │
  │               │◄─409 Conflict──│                  │              │
  │◄─409──────────│                │                  │              │
```

**Steps:**
1. Client sends registration payload (first name, last name, email, password).
2. API delegates to the Facade; the Facade checks for duplicate email.
3. If no duplicate, a `User` object is created with the password hashed.
4. The model persists via `INSERT` and returns the saved row.
5. The Facade serializes with `to_dict()` and the API responds 201 Created.
6. If the email is already registered, the Facade raises a conflict and the API returns 409.

### 3.2 Place Creation

**Endpoint:** `POST /api/places` *(requires authentication)*

```
Client          API              Facade           Place model        DB
  │               │                │                  │               │
  │─POST /api/places + JWT────────►│                  │               │
  │               │ verify JWT →   │                  │               │
  │               │   user_id      │                  │               │
  │               │─create_place(data, owner_id)────►│               │
  │               │                │ validate lat/lng │               │
  │               │                │─Place(...)───────────────────── ►│
  │               │                │                  │─INSERT──────► │
  │               │                │                  │◄──place_id── │
  │               │                │◄──place.to_dict()─────────────── │
  │               │◄──{id,title,…}─│                  │               │
  │◄─201 Created──│                │                  │               │
  │               │                │                  │               │
  │[alt] invalid or missing token   │                  │               │
  │◄─401 Unauthorized──────────────│                  │               │
```

**Steps:**
1. Client sends place data with a JWT in the `Authorization` header.
2. API verifies the token and extracts `user_id` (the owner).
3. Facade validates business rules (price > 0, latitude −90..90, longitude −180..180).
4. A `Place` object is created, saved, and returned as 201 Created.
5. Expired or missing tokens result in 401 Unauthorized without touching the business logic.

### 3.3 Review Submission

**Endpoint:** `POST /api/places/{place_id}/reviews` *(requires authentication)*

```
Client          API              Facade           Review model       DB
  │               │                │                  │               │
  │─POST /api/places/{id}/reviews─►│                  │               │
  │               │ verify JWT     │                  │               │
  │               │─submit_review(place_id, uid, data)►              │
  │               │                │─SELECT place WHERE id=…────────►│
  │               │                │◄───────────────── place ────────│
  │               │                │─SELECT review WHERE place+user─►│
  │               │                │◄──────────────── None ──────────│
  │               │                │─Review(rating, comment, …)──── ►│
  │               │                │                  │─INSERT──────►│
  │               │                │                  │◄──OK─────── │
  │               │                │◄─review.to_dict()────────────── │
  │               │◄──{id, rating}─│                  │               │
  │◄─201 Created──│                │                  │               │
  │               │                │                  │               │
  │[alt] user already reviewed this place              │               │
  │◄─409 Conflict──────────────────│                  │               │
```

**Steps:**
1. Facade first confirms the target place exists (404 if not).
2. Facade checks that this user has not already reviewed this place (409 if they have).
3. Rating must be in range 1–5 (422 Unprocessable Entity if not).
4. `Review` object is created, persisted, and the 201 response is sent.

### 3.4 Fetch List of Places

**Endpoint:** `GET /api/places?city=Paris&max_price=100` *(public)*

```
Client          API              Facade           Place model        DB
  │               │                │                  │               │
  │─GET /api/places?city=…────────►│                  │               │
  │               │ parse params   │                  │               │
  │               │─get_places(filters)─────────────►│               │
  │               │                │ build query      │               │
  │               │                │─Place.list(filters)──────────── ►│
  │               │                │                  │─SELECT…──── ►│
  │               │                │                  │◄──rows[]─── │
  │               │                │                  │ serialize    │
  │               │                │◄──[{place},…]─────────────────  │
  │               │◄──[{place},…]──│                  │               │
  │◄─200 OK───────│                │                  │               │
  │               │                │                  │               │
  │[alt] no results matching filters                   │               │
  │◄─200 OK []────│                │                  │               │
```

**Steps:**
1. Query parameters are parsed into a filter object (city, max_price, min_price, amenity IDs, etc.).
2. Facade builds the database query dynamically based on present filters.
3. `Place.list(filters)` executes a filtered SELECT and returns ORM objects.
4. Each place is serialized via `to_dict()` before the list is returned.
5. An empty result set returns 200 OK with an empty array (not 404).

## 4. Design Decisions

**Why a Facade?** The Facade consolidates all cross-cutting logic (authorization checks, duplicate detection, validation ordering) in one place. Without it, presentation-layer code would need to know the internal structure of models, and tests would have to mock HTTP contexts.

**Why `BaseModel`?** Centralizing `id`, `created_at`, and `updated_at` in a single parent class ensures every entity is automatically auditable and consistently identifiable. The `save()` hook gives a single place to add pre-persist logic (e.g. updating `updated_at`).

**UUID vs. integer IDs.** UUIDs are used for primary keys to allow future distribution (sharding, replication) without ID collisions, and to prevent enumeration attacks in API URLs.

**Password handling.** Passwords are never stored in plain text. The `User.register()` method hashes the password (bcrypt or Argon2) before calling `save()`. The plain password is never stored in any attribute.

**Many-to-many (Place ↔ Amenity).** A join table `place_amenities` will be introduced in Part 3. In the Business Logic layer, this is modeled as `Place.amenities: list[Amenity]`.

**HTTP status codes used:**
- 200 OK — successful GET
- 201 Created — successful POST
- 401 Unauthorized — missing or invalid token
- 404 Not Found — resource does not exist
- 409 Conflict — duplicate email, duplicate review
- 422 Unprocessable Entity — invalid field values (bad rating, out-of-range coordinates)
