# HBnB Evolution — Sequence Diagrams for API Calls

Four sequence diagrams covering the primary API interactions in the HBnB Evolution application. Each diagram traces a request from the client through the Presentation Layer, Business Logic Layer, and Persistence Layer, then back.

---

## Notation key

| Symbol | Meaning |
|---|---|
| `->>` | Synchronous call (solid arrow) |
| `-->>` | Return value (dashed arrow) |
| `Note` | Inline annotation |
| `alt / else` | Conditional branch |

---

## 1. User Registration

**Endpoint:** `POST /api/users`  
**Auth required:** No  
**Purpose:** A new user submits their details. The Facade checks for a duplicate email, hashes the password, persists the user, and returns the created resource.

**Key steps:**
1. Client sends `{first_name, last_name, email, password}`.
2. API passes the payload to the Facade via `validate_and_register()`.
3. Facade queries the DB to check whether the email is already registered.
4. If unique, a `User` object is instantiated with a hashed password and saved.
5. The saved record is serialized with `to_dict()` and returned as `201 Created`.
6. If the email exists, the Facade raises a conflict and the API returns `409 Conflict`.

```mermaid
sequenceDiagram
    participant Client
    participant API as API (Presentation)
    participant Facade as Facade (Business Logic)
    participant UserModel as User Model
    participant DB as Database (Persistence)

    Client->>API: POST /api/users {email, password, first_name, last_name}
    API->>Facade: validate_and_register(data)
    Facade->>DB: find_by_email(email)
    DB-->>Facade: None (no duplicate found)
    Facade->>UserModel: User(data) — hash password
    UserModel->>DB: INSERT INTO users
    DB-->>UserModel: OK / row id
    UserModel-->>Facade: user.to_dict()
    Facade-->>API: {id, email, created_at}
    API-->>Client: 201 Created {user}

    alt Email already registered
        Facade-->>API: 409 Conflict
        API-->>Client: 409 {error: email_taken}
    end
```

## 2. Place Creation

**Endpoint:** `POST /api/places`  
**Auth required:** Yes (JWT)  
**Purpose:** An authenticated user creates a property listing. The API verifies the token to extract the owner's identity, the Facade validates the payload, and the new place is persisted.

**Key steps:**
1. Client sends place data with a JWT in the `Authorization` header.
2. API verifies the token and extracts `owner_id`.
3. Facade validates business rules: `price > 0`, `latitude ∈ [−90, 90]`, `longitude ∈ [−180, 180]`.
4. A `Place` object is created, saved, and serialized.
5. Response is `201 Created` with the place payload.
6. An invalid or expired token returns `401 Unauthorized` immediately.

```mermaid
sequenceDiagram
    participant Client
    participant API as API (Presentation)
    participant Facade as Facade (Business Logic)
    participant PlaceModel as Place Model
    participant DB as Database (Persistence)

    Client->>API: POST /api/places {title, price, lat, lng} + JWT
    Note over API: verify JWT → extract owner_id
    API->>Facade: create_place(data, owner_id)
    Note over Facade: validate price > 0, lat ±90, lng ±180
    Facade->>PlaceModel: Place(title, price, lat, lng, owner)
    PlaceModel->>DB: INSERT INTO places
    DB-->>PlaceModel: place_id
    PlaceModel-->>Facade: place.to_dict()
    Facade-->>API: {id, title, price, lat, lng}
    API-->>Client: 201 Created {place}

    alt Missing or expired token
        API-->>Client: 401 Unauthorized
    end
```

## 3. Review Submission

**Endpoint:** `POST /api/places/{place_id}/reviews`  
**Auth required:** Yes (JWT)  
**Purpose:** An authenticated user submits a rating and comment for a place. The Facade enforces two business rules before persisting: the place must exist, and the user must not have already reviewed it.

**Key steps:**
1. Client sends `{rating, comment}` with a JWT for the target place.
2. API verifies the token and extracts `user_id`.
3. Facade queries the DB to confirm the place exists (`404 Not Found` if not).
4. Facade queries for an existing review by the same user for the same place (`409 Conflict` if found).
5. A `Review` object is created with a validated rating (1–5) and persisted.
6. Response is `201 Created` with the review payload.

```mermaid
sequenceDiagram
    participant Client
    participant API as API (Presentation)
    participant Facade as Facade (Business Logic)
    participant ReviewModel as Review Model
    participant DB as Database (Persistence)

    Client->>API: POST /api/places/{id}/reviews {rating, comment} + JWT
    Note over API: verify JWT → extract user_id
    API->>Facade: submit_review(place_id, user_id, data)
    Facade->>DB: SELECT * FROM places WHERE id = place_id
    DB-->>Facade: place row
    Facade->>DB: SELECT * FROM reviews WHERE place_id AND user_id
    DB-->>Facade: None (no duplicate review)
    Facade->>ReviewModel: Review(rating, comment, place, user)
    ReviewModel->>DB: INSERT INTO reviews
    DB-->>ReviewModel: OK
    ReviewModel-->>Facade: review.to_dict()
    Facade-->>API: {id, rating, comment}
    API-->>Client: 201 Created {review}

    alt Place not found
        Facade-->>API: 404 Not Found
        API-->>Client: 404 {error: place_not_found}
    else User already reviewed this place
        Facade-->>API: 409 Conflict
        API-->>Client: 409 {error: duplicate_review}
    end
```

## 4. Fetching a List of Places

**Endpoint:** `GET /api/places`  
**Auth required:** No  
**Purpose:** Returns a filtered, paginated list of place listings. The client may pass optional query parameters. The Facade translates these into a dynamic database query and returns the results.

**Key steps:**
1. Client sends a `GET` request with optional filters (e.g. `?city=Paris&max_price=100`).
2. API parses the query string into a structured filter object.
3. Facade constructs a dynamic `SELECT` query from the filters.
4. `Place.list(filters)` runs the query and returns ORM objects.
5. Each place is serialized with `to_dict()` before the list is assembled.
6. Response is `200 OK` with a JSON array (empty array if no results — not 404).

```mermaid
sequenceDiagram
    participant Client
    participant API as API (Presentation)
    participant Facade as Facade (Business Logic)
    participant PlaceModel as Place Model
    participant DB as Database (Persistence)

    Client->>API: GET /api/places?city=Paris&max_price=100
    Note over API: parse query params → filters object
    API->>Facade: get_places(filters)
    Note over Facade: build dynamic filtered query
    Facade->>PlaceModel: Place.list(filters)
    PlaceModel->>DB: SELECT * FROM places WHERE ...
    DB-->>PlaceModel: rows[]
    Note over PlaceModel: [p.to_dict() for p in rows]
    PlaceModel-->>Facade: [{place}, ...]
    Facade-->>API: [{place}, ...]
    API-->>Client: 200 OK [{places}]

    alt No results match filters
        API-->>Client: 200 OK []
    end
```

## Layer contribution summary

| Layer | Role in each request |
|---|---|
| **Presentation (API)** | Receives HTTP request, parses input, authenticates tokens, serializes the final HTTP response |
| **Business Logic (Facade + Models)** | Enforces domain rules (uniqueness, ownership, rating range, coordinate bounds), orchestrates model creation, calls persistence |
| **Persistence (DB)** | Executes SQL, returns raw rows; never aware of HTTP or business rules |
